# CRISIS LEADERSHIP — COMMAND AUTHORITY PROTOCOL

**(A+ Standard — Institutional Command Architecture)**

---

## 1. Executive Summary

The single most critical decision in any crisis is not what to do — it is **who decides**.

In every major institutional failure, the root cause is not a lack of intelligence or capability. It is a breakdown in command authority. Either no one knew who was in charge, or the person in charge did not have the authority to execute.

This protocol provides:

```txt
Command Authority Framework — who commands, for what duration, with what scope
Authority Transfer Protocol — how command moves between phases and people
Decision Rights Matrix — what each role can decide without escalation
Command Log Architecture — real-time decision recording for post-crisis review
```

It is designed to be pre-installed and activated within minutes of crisis onset.

---

## 2. Strategic Context

Command authority fails in five predictable patterns:

```txt
1. Vacuum — no one is designated to command. Decisions stall.
2. Contest — multiple people believe they command. Decisions conflict.
3. Scope creep — commander exceeds their authority. Decisions are invalidated.
4. Fatigue — commander burns out but no transfer protocol exists. Decisions degrade.
5. Abandonment — commander withdraws without handover. Decisions stop.
```

Each pattern has a structural cause. Each has a structural correction.

### The Command Spectrum

```txt
Peace             → Distributed authority, consensus-based
Crisis (Level 1)  → Centralised authority, commander-led
Crisis (Level 2)  → Single command, no appeals
Crisis (Level 3)  → Single command, override authority activated
Recovery          → Gradual return to distributed authority
```

The level of centralisation must match the severity of the crisis. Over-centralisation creates bottlenecks. Under-centralisation creates chaos.

## 3. Core Model — Command Authority Framework

### 3.1 The Three Command Levels

```txt
Level 1 — Tactical Command
  Scope: Operational response within defined parameters
  Authority: Execute pre-approved playbooks
  Duration: 24 hours, renewable
  Example: Incident commander for operational disruption

Level 2 — Strategic Command
  Scope: Cross-functional crisis response, resource allocation
  Authority: Override normal governance, commit resources up to defined limit
  Duration: 72 hours, renewable by board
  Example: Crisis commander for revenue shock or regulatory intervention

Level 3 — Sovereign Command
  Scope: Institutional survival, existential threat
  Authority: Full override of normal governance, unlimited resource commitment
  Duration: 7 days, renewable by board with supermajority
  Example: Commander for existential threat to the institution
```

### 3.2 Command Authority Matrix

```txt
Crisis Type          | Command Level | Commander        | Board Role
─────────────────────┼───────────────┼─────────────────┼─────────────────────
Operational Breakdown| Level 1       | COO/Operations   | Notified
Revenue Shock        | Level 2       | CEO              | Approve strategy
Leadership Loss      | Level 2       | Board Chair      | Activate succession
Regulatory Action    | Level 2       | General Counsel  | Oversee response
Reputational Crisis  | Level 2       | CEO              | Approve narrative
Existential Threat   | Level 3       | Board Chair/CEO  | Supermajority support
```

### 3.3 Command Authority Boundaries

```txt
Every commander operates within defined boundaries:

Financial Authority:   Maximum commitment without escalation (£)
Personnel Authority:   Hiring, firing, reassignment scope
Communication Authority: What can be said, to whom, without clearance
Strategic Authority:   What decisions can be made without board approval

Violating these boundaries invalidates the decision and triggers escalation.
```

## 4. Authority Transfer Protocol

### 4.1 When Transfer Is Required

```txt
Phase transition:     Moving from containment to stabilisation to recovery
Commander fatigue:    After 24 hours at Level 2, 72 hours at Level 3
Scope expansion:      Crisis exceeds original command authority
Commander incapacity: Physical or mental inability to continue
Conflict emergence:   Dispute over command authority that cannot be resolved
```

### 4.2 Transfer Sequence

```txt
Step 1: Declare transfer intent (commander + board chair)
Step 2: Brief incoming commander on situation, decisions, and pending items
Step 3: Transfer authority symbolically (written order, verbal confirmation)
Step 4: Communicate transfer to all stakeholders (internal first, then external)
Step 5: Outgoing commander remains available for consultation (not decision)
Step 6: Document transfer in command log with timestamp and rationale
```

### 4.3 Transfer Anti-Patterns

```txt
Shadow command — outgoing commander continues making decisions informally
Premature transfer — command transferred before incoming commander is ready
Unilateral transfer — commander transfers without board notification
Non-transfer — commander refuses to transfer despite fatigue or incapacity
```

## 5. Decision Rights Matrix

### 5.1 Decision Classification Under Command

```txt
Class 1 — Commander decides alone
  Criteria: Time-sensitive, within authority boundaries, pre-approved playbook
  Examples: Tactical resource allocation, communication within approved narrative

Class 2 — Commander decides after consultation
  Criteria: Significant impact, some time available, novel situation
  Examples: Major resource commitment, strategy adjustment, stakeholder engagement

Class 3 — Commander recommends, board decides
  Criteria: Existential impact, irreversible, exceeds commander's authority
  Examples: Merger/acquisition, leadership succession, institutional restructuring
```

### 5.2 Escalation Rules

```txt
Any decision that:
  - Exceeds financial authority
  - Creates irreversible institutional consequence
  - Requires override of constitutional governance
  - Affects the institution's legal or regulatory standing

Must be escalated to the next command level or the board.
```

## 6. Command Log Architecture

### 6.1 Purpose

The command log is the institution's legal and operational record of crisis decisions. It serves three functions:

```txt
1. Accountability — every decision has an owner, rationale, and timestamp
2. Continuity — incoming commander can review all decisions made to date
3. Learning — post-crisis review has complete decision data
```

### 6.2 Log Entry Format

```txt
Timestamp:     [ISO 8601]
Decision:      [What was decided]
Owner:         [Who decided]
Class:         [1/2/3]
Authority:     [Within/Exceeds boundaries]
Rationale:     [Why this decision]
Evidence:      [What information supported the decision]
Outcome:       [What happened as a result]
Review Flag:   [Requires post-crisis review: Yes/No]
```

### 6.3 Log Discipline

```txt
Every decision must be logged within 15 minutes.
No exception. No retrospective logging.
If it was not logged, it was not decided.
```

## 7. Diagnostic Signals

Signals that command authority is failing or absent:

```txt
COMMAND_VACUUM         — no one is making decisions; issues pile up
COMMAND_CONTEST        — multiple people claiming authority; decisions conflict
COMMAND_FATIGUE        — commander showing signs of overload; decision quality dropping
COMMAND_SCOPE_VIOLATION — commander operating outside authority boundaries
COMMAND_ISOLATION      — commander not consulting necessary expertise
COMMAND_SILENCE        — commander not communicating decisions to stakeholders
```

---

## 8. Failure Modes

```txt
1. Democratic crisis management — treating crisis as a consensus exercise. Crisis demands command, not committee.

2. Heroic commander — single person carrying the entire response. Works once. Fails when that person is unavailable.

3. Frozen commander — commander cannot decide under pressure. No transfer protocol exists to replace them.

4. Rogue commander — commander exceeds authority boundaries. No mechanism to recall or override them.

5. Shadow commander — previous commander continues making decisions informally. Undermines current command.

6. Exhausted commander — commander operates beyond sustainable duration. Decision quality degrades silently.
```

## 9. Application Playbook

### Step 1 — Pre-Designate Commanders

For each crisis type in the institution's risk register, name the commander, their authority boundaries, and their duration limit.

**Deliverable:** Crisis Command Authority Register

### Step 2 — Install Command Log System

Create the command log template. Train senior leaders on its use. Test under simulation.

**Deliverable:** Command Log Template + Training Completion

### Step 3 — Define Escalation Thresholds

For each command level, define the specific thresholds that trigger escalation to the next level.

**Deliverable:** Escalation Threshold Document

### Step 4 — Practice Authority Transfer

Run a simulation that requires command transfer mid-crisis. Test the transfer protocol.

**Deliverable:** Transfer Simulation Report

### Step 5 — Install Post-Crisis Review

Define the post-crisis review process. Include command log analysis as a mandatory input.

**Deliverable:** Post-Crisis Review Protocol

## 12. Case Application (Modelled)

### Situation

Financial services firm. Regulatory investigation announced. CEO is designated Level 2 commander. Investigation expands to include CEO's prior decisions.

### Crisis

Command conflict emerges. CEO cannot command the response to an investigation that includes their own decisions.

### Protocol Activated

```txt
Command transfer triggered (scope expansion + conflict emergence)
Board Chair assumes Level 2 command
General Counsel designated tactical commander for regulatory response
CEO recused from all investigation-related decisions
Transfer logged, communicated, and documented
```

### Outcome

```txt
Regulatory response led by unconflicted commander
CEO able to focus on business continuity
Investigation handled without command interference
Post-crisis review confirmed transfer was correct and timely
```

## 15. Final Position

Crisis command is not about authority. It is about clarity.

When everyone knows who decides, decisions happen faster, better, and with less friction.

When no one knows, the crisis decides for you.

Install the command protocol before the crisis arrives. The crisis will not wait for you to figure it out.

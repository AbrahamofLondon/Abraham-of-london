# Diagnostic Engine Architecture

Technical reference for the Abraham of London diagnostic decision engine.

## Input Sources

The engine ingests three categories of input:

1. **User answers** — structured responses from the diagnostic questionnaire (urgency, ownership, state, clarity, accountability scores)
2. **Respondent data** — when multiple respondents from the same organisation complete assessments, enabling cross-respondent collision detection
3. **Evidence nodes** — persisted `DiagnosticEvidenceNode` records (kind, label, summary, confidence, severity, payload) that accumulate across sessions

## Scoring Models — 6 Archetypes

The decision engine (`lib/diagnostics/decision-engine.ts`) classifies every decision into one of six fixed archetypes:

| Archetype | Core Decision |
|-----------|---------------|
| `mandate_vacuum` | Define authority versus delay |
| `reactive_decision_pattern` | Commit versus defer |
| `structural_inconsistency` | Standardise versus fragment |
| `trust_asymmetry` | Centralise trust repair versus distribute assumptions |
| `execution_drift` | Stabilise ownership versus absorb drift |
| `cost_blindness` | Quantify cost versus absorb unpriced loss |

Each archetype carries a fixed `defaultConsequence` and `defaultAction`. Classification is deterministic based on signal weights and severity scores — not generated by AI.

## Deterministic Rules

### Contradiction Detection

`lib/diagnostics/contradictions.ts` implements three named contradiction types:

- **URGENCY_VS_OWNERSHIP** — triggered when urgency >= 3 AND ownership score >= 3. Indicates decisions being made by whoever acts first rather than designated authority.
- **CLARITY_VS_ACCOUNTABILITY** — triggered when clarity >= 3 AND accountability >= 3. A defined decision without accountability fails during execution.
- **URGENCY_VS_STATE** — triggered when urgency >= 3 AND state score >= 3. Urgent decisions that are repeatedly deferred become more complex.

These fire deterministically from numeric inputs. No AI inference is involved.

### Severity Classification

Evidence nodes carry severity levels (`low`, `medium`, `high`, `critical`). The signal detector and evidence graph compute severity from:
- Score thresholds on input dimensions
- Cross-respondent divergence magnitude
- Time-based escalation (longer delay = higher severity)

## AI Use Boundaries

The system draws a hard line between deterministic computation and AI-generated content:

| Layer | Method | Rejectable? |
|-------|--------|-------------|
| Classification (archetype) | Deterministic | N/A — fixed |
| Contradiction detection | Deterministic | N/A — fixed |
| Confidence scoring | Deterministic | N/A — computed |
| Synthesis narrative | AI-generated | YES — arbiter can reject |
| Concrete move language | AI-generated | YES — arbiter validates |

AI generates the user-facing synthesis text. The arbiter tournament validates that synthesis against deterministic outputs. If they disagree, the synthesis is rejected and the user sees the mismatch explicitly.

## Arbiter Tournament Logic

`lib/decision/arbiter-tournament.ts` implements 5 mandatory validation rules. Any hard violation rejects the synthesis entirely.

### Rule 1: Condition Integrity
Synthesis condition class must match the deterministic classification. If the AI says "this is execution drift" but the deterministic engine classified it as "mandate vacuum", the synthesis is rejected.

### Rule 2: Contradiction Alignment
Synthesis must reference terms from the deterministic contradiction set. The AI cannot invent contradictions that the scoring engine did not detect.

### Rule 3: Move Validity
The concrete move must reference the actual blocker identified by the spine, not generic advice.

### Rule 4: Cost Consistency
No invented costs are permitted without a user-stated cost input. The AI cannot fabricate financial figures.

### Rule 5: Avoidance Proof
Synthesis must reference either the forced action or a prior-attempt mismatch. It cannot soft-pedal by avoiding the uncomfortable truth.

Hard violations trigger immediate rejection. Soft violations issue a warning but allow the synthesis through. The user always sees the mismatch count and the weakest domain.

## Confidence Scoring

Confidence operates on a `low | medium | high` scale with specific meanings:

- **low** — insufficient signal density; single respondent, few evidence nodes
- **medium** — adequate signal but missing cross-respondent validation or outcome verification
- **high** — multiple respondents, cross-validated signals, or verified prior outcomes

The numerical confidence on evidence nodes ranges from 0 to 1.0 (stored as Float). The system caps maximum confidence at 0.95 — no assessment claims certainty.

Respondent count directly affects confidence: a single-respondent assessment cannot reach `high` confidence regardless of signal clarity.

## Accuracy Feedback Loop

### Pressure Loop (48h / 7d / 14d)

`lib/follow-up/pressure-loop.ts` creates a three-stage follow-up sequence after each diagnostic:

- **T+48h** — "You said you would do this." Direct recall of the stated action. No encouragement.
- **T+7d** — "This is now a pattern." Escalation noting that the cost of delay has compounded.
- **T+14d** — "The system has updated your condition." Final notice that the diagnostic now includes avoidance as a signal.

Messages are idempotent (keyed to prevent duplicates). Action recording stops the loop.

### Escalation Engine (24h / 48h / 72h / 5d / 7d)

`lib/follow-up/escalation-engine.ts` segments users into four categories:

- **A** — Intent confirmed, no purchase. High-pressure conversion.
- **B** — No intent. Challenge + escalation.
- **C** — Low cost (<5k). Educate or discard.
- **D** — High cost (>20k). Immediate Strategy Room escalation.

Five-stage messaging with tones progressing from `reality` to `fork`.

### Decision Credit Score

`lib/follow-up/decision-credit-score.ts` computes a 0-100 behavioural reputation metric from four components:

| Component | Range | Measures |
|-----------|-------|----------|
| `followThrough` | 0-30 | Ratio of commitments where action was taken |
| `breachPenalty` | 0-25 (subtracted) | Accumulated checkpoint misses |
| `impactQuality` | 0-25 | Proportion of verified structural changes |
| `consistency` | 0-20 | Integrity score stability over time |

Trend is computed by comparing current score to previous score. Delta > 5 = improving; delta < -5 = declining.

## Known Limitations

1. **Single-respondent ceiling** — A single respondent cannot produce `high` confidence. The system explicitly states when cross-validation would change the assessment.

2. **AI synthesis rejection** — When the arbiter rejects synthesis, the user sees raw deterministic output. This is intentional (transparency over polish) but produces less readable reports.

3. **Cold-start scoring** — New users with no prior spines receive a default credit score of ~50. The score becomes meaningful only after 2-3 interactions.

4. **Contradiction detection coverage** — Only 3 contradiction types are currently implemented. Complex multi-dimensional contradictions (e.g., trust asymmetry combined with execution drift) are not yet detected as compound patterns.

5. **Pressure loop dependency on email delivery** — The feedback loop assumes email reach. If emails bounce or are filtered, the pressure signal cannot fire and the system cannot record the absence of action.

6. **Outcome verification lag** — Outcome classification requires a follow-up assessment. The minimum meaningful verification window is approximately 14 days, creating a measurement gap where the system cannot confirm movement.
